PUT /api/programs/:id/assignments uses delete-all-then-reinsert (not incremental upsert)
AUDIT
Origin: Observed May 11 during B1 setup via direct curl. PUT request with full assignment list returned new IDs for ALL rows (not just the newly-added SHK row) — confirming the endpoint deletes everything and reinserts rather than reconciling diffs. Impact: (a) Any future FK from another table referencing assignments.id would orphan on every PUT. Currently FK enforcement is OFF, so this is latent — but flipping FK enforcement ON during Azure SQL migration without addressing this could break assignment writes silently. (b) Any historical audit trail tied to assignment IDs is lost on every save. (c) Performance: scales as O(N) on N TCs per program — fine at current scale, but worth noting. Fix scope: Refactor to UPSERT pattern: compute diff between existing and incoming, INSERT new + UPDATE changed + DELETE removed. Heavier change (~50-80 LOC) and would benefit from test coverage before shipping. Why not now: Not Stage A blocker; current scale and lack of FK enforcement mean no user-visible symptom. Track for Azure SQL migration window where FK enforcement decision happens.

Authorization

Threat model

SSO confirms identity; code enforces permissions (different concerns)
